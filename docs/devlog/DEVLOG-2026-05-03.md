> Part 1: ScanPlainHelmReleases via helm.sh/helm/v3/pkg/action.

## Files touched

- `internal/scanner/cluster.go` — complete rewrite; added RepoURL field, scanFluxHelmReleases, scanPlainHelmReleases, scanArgoCDApplications, ScanAll
- `internal/controller/compatibilitypolicy_controller.go` — updated call site from ScanHelmReleases → ScanAll
- `go.mod` / `go.sum` — added helm.sh/helm/v3 v3.16.4, k8s.io/cli-runtime v0.31.3 as direct deps; go mod tidy pulled transitive deps

## Key decisions

- Used `genericclioptions.NewConfigFlags(true)` per spec; `action.Configuration.Init` returns error in v3.16.4 so it is captured.
- Helm SDK (v3.16.4) pinned because later versions (v3.20.x) had not been validated against the k8s v0.35.0 deps already in the module.
- `k8s.io/cli-runtime` came in as a transitive dep of helm at v0.31.3; made it explicit so it survives `go mod tidy`.
- ArgoCD scanner uses unstructured + apimeta.IsNoMatchError to silently skip clusters where the CRD is absent — same pattern as FluxCD scanner.
- ScanAll logs individual scanner errors with controller-runtime logger and continues; it never propagates an error from one source to the caller.
- Namespace access-denied in the Helm scanner is treated as a per-namespace skip (logged at Info), not a fatal error.

# DEVLOG — 2026-05-03 — Session 11: Compatibility Matrix planner + knowledge YAML base

## Goal

Build `internal/planner/matrix.go` — a YAML-driven compatibility matrix that loads per-tool knowledge files and answers upgrade-resolution queries — and populate the knowledge base with entries for all six tracked tools.

## Prompt Used

> Build the compatibility matrix planner: Load() reads all *.yaml files from a directory into a Matrix. Resolve(tool, fromVersion, toVersion) returns a CompatibilityEntry. LatestVersion(tool) returns the last version. Write table-driven tests against the real knowledge/tools directory. Add knowledge YAML files for argo-cd, external-secrets, istio, prometheus-stack, and vault (cert-manager already existed).

## Files Touched

- `internal/planner/matrix.go` — Load() rewritten to accept a directory and glob `*.yaml`; duplicate tool guard added
- `internal/planner/matrix_test.go` — helper renamed to `knowledgeToolsDir`; `TestAllToolsLoaded` added; `TestLoad_DuplicateTool` added
- `knowledge/tools/argo-cd.yaml` — new; 6 versions (2.7–2.12), incompatibilities with ingress-nginx, breaking changes
- `knowledge/tools/external-secrets.yaml` — new; 5 versions (0.8–0.10.5), store API and ClusterSecretStore breaking changes
- `knowledge/tools/istio.yaml` — new; 5 versions (1.18–1.22), sidecar-restart breaking change on every minor bump, cert-manager ACME incompatibilities
- `knowledge/tools/prometheus-stack.yaml` — new; 6 versions (45–62), CRD migration and Grafana subchart breaking changes
- `knowledge/tools/vault.yaml` — new; 5 versions (0.24–0.28), Raft config and injector breaking changes

## What Was Built

- `Load(dir)` globs `*.yaml`, unmarshals each into `ToolCompatibility`, rejects missing `tool` fields and duplicate tool names, errors on empty directory
- `Resolve(tool, from, to)` walks the version list and returns a `CompatibilityEntry` embedding the target `VersionEntry` alongside tool name and source version
- `LatestVersion(tool)` returns the last entry in the versions slice
- Full YAML knowledge base for 5 new tools; each tool has ≥3 versions, all versions carry breaking changes, risk levels, min_kubernetes, incompatible_with entries, and upgrade notes
- Test suite: 13 tests covering happy-path load, missing dir, missing tool field, duplicate tool detection, unknown tool, unknown version, all-tools validation (min 3 versions + ≥1 breaking change each), and 4 specific cert-manager upgrade paths with exact field assertions

## Key Decisions

- `Load` returns an error on empty directory rather than an empty Matrix, to catch misconfigured deployments early.
- Duplicate tool detection prevents silent shadowing when two YAML files declare the same tool name.
- `LatestVersion` reads the last slice element (file-order); no semver sorting — callers are expected to order versions ascending in YAML, which is the convention enforced by the knowledge files.
- YAML knowledge files use inline comments to explain why specific incompatibilities exist, making the database self-documenting.
